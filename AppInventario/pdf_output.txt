

 
Instrucción  
 
El presente documento describe el diseño arquitectónico y el modelo de diseño 
orientado  a  objetos para  un  Sistema de  Inventario  pensado  para  empresas 
medianas con múltiples almacenes, control de lotes y necesidad de integración 
con puntos de venta y sistemas contables. Se parte de los fundamentos de la 
arquitectura  de  software  y  del  análisis  orientado  a  objetos para  transformar 
requisitos reales de usuarios en una solución coherente, modular y preparada 
para la evolución tecnológica. 
El  objetivo  es  ofrecer  un anteproyecto  técnico que  sirva  como guía  para  la 
construcción: especifica los requisitos funcionales y no funcionales, propone una 
arquitectura  híbrida  cliente-servidor  con  servicios  modulares  y  describe  las 
principales entidades del dominio mediante diagramas y modelos de clases. Este 
enfoque  busca  equilibrar  la  resolución  de  las  necesidades  inmediatas  del 
negocio  con  la  flexibilidad  necesaria  para  adaptarse  a  cambios  futuros, 
garantizando mantenibilidad, seguridad y escalabilidad. 
El documento incluye además ejemplos de interacción (casos de uso), patrones 
de diseño orientado a objetos aplicables y consideraciones para una posible 
migración a microservicios. Está pensado para ser utilizado por desarrolladores, 
arquitectos y responsables de producto como base para la implementación, las 
pruebas y la planificación de despliegues. 
 
 
 
 
 
 
 
 

 
 
Sistema de Inventario 
Descripción breve Aplicación para gestionar productos, existencias, entradas y 
salidas, proveedores, órdenes de compra y reportes en tiempo real para una o 
varias bodegas. 
 Dominio de aplicación y alcance 
Dominio elegido Sistema de Inventario para empresas medianas que manejan 
múltiples  almacenes,  control  de  lotes  y  fechas  de  vencimiento,  y  requieren 
integración con punto de venta y compras. 
Alcance funcional mínimo entregable 
✓ Gestión de productos y categorías 
✓ Control de existencias por almacén y por lote 
✓ Registro de entradas y salidas (recepciones, ventas, ajustes) 
✓ Gestión de proveedores y órdenes de compra 
✓ Generación de reportes e indicadores (stock, rotación, alertas de bajo 
stock) 
✓ Autenticación y control de roles y permisos 
✓ API para integración con POS y sistemas contables

debe cumplir. Incluye las funciones principales, el rendimiento esperado, 
la seguridad, la escalabilidad y otros aspectos relevantes. 
 
Requisitos funcionales principales 
1. RF01 Registrar producto: Crear/editar/eliminar productos con código, 
descripción, unidad, categoría, precio, proveedor preferente, lote y fecha 
de vencimiento. 
2. RF02 Gestión de almacenes: Crear múltiples almacenes; consultar stock 
por almacén. 
3. RF03  Movimientos  de  inventario:  Registrar  entradas  (recepciones), 
salidas (ventas, transferencias), ajustes y devoluciones. 

4. RF04  Órdenes  de  compra:  Generar,  aprobar  y  recibir  órdenes  de 
compra; actualizar stock al recibir. 
5. RF05  Transferencias  entre  almacenes:  Solicitar  y  completar 
transferencias internas. 
6. RF06 Alertas y notificaciones: Alertas por bajo stock, vencimiento de 
lotes y recepción pendiente. 
7. RF07 Reportes: Reportes de stock, historial de movimientos, rotación de 
inventario, valor de inventario. 
8. RF08  Autenticación  y  autorización:  Login,  roles  (admin,  almacén, 
comprador, auditor) y permisos por acción. 
9. RF09 API REST: Endpoints para consultar stock, registrar movimientos y 
sincronizar con POS. 
10. RF10 Auditoría: Registro inmutable de transacciones de inventario con 
usuario y timestamp. 
Requisitos no funcionales 
1. Rendimiento:  Respuesta  <  300  ms  para  consultas  de  stock  en 
condiciones normales; capacidad para 200 transacciones concurrentes en 
picos. 
2. Disponibilidad:  99.9%  SLA  para  la  API  crítica;  ventanas  de 
mantenimiento planificadas. 
3. Escalabilidad: Escalado horizontal de la capa de aplicación y de la base 
de datos mediante réplicas y particionado por almacén si es necesario. 
4. Seguridad: Autenticación basada en tokens JWT; cifrado TLS en tránsito; 
cifrado de datos sensibles en reposo; control de acceso por roles. 
5. Consistencia: Consistencia eventual aceptable para reportes agregados; 
consistencia fuerte para operaciones de decremento de stock (reservas y 
ventas). 
6. Mantenibilidad:  Código  modular,  pruebas  unitarias  y  de  integración, 
documentación de API y diagramas de arquitectura. 
7. Portabilidad: Contenedores Docker para despliegue en nubes públicas o 
privadas. 
8. Observabilidad:  Logs  estructurados,  métricas  (Prometheus)  y  trazas 
distribuidas (OpenTelemetry). 
9. Internacionalización:  Soporte  para  múltiples  idiomas  y  formatos  de 
fecha/moneda. 

10. Backup y recuperación: Backups diarios y plan de recuperación con 
RTO < 2 horas y RPO < 1 hora para datos críticos.

3 Diseño de arquitectura propuesto 
Elección de estilo arquitectónico 
Arquitectura híbrida cliente servidor con servicios modulares 
✓ Cliente: Aplicación web SPA (React o Angular) y cliente móvil ligero. 
✓ Servidor: Backend RESTful monolítico modular o conjunto de servicios 
desacoplados (híbrido preparado para migrar a microservicios). 
✓ Persistencia: Base de datos relacional (PostgreSQL) para transacciones; 
caché Redis para consultas frecuentes; almacenamiento de archivos en 
S3 compatible. 
✓ Integraciones: API Gateway para exponer endpoints; colas (RabbitMQ o 
Kafka)  para  procesos  asíncronos  (notificaciones,  sincronización  con 
POS).

Flujo simplificado 
1. Usuario realiza acción en Cliente. 
2. Cliente llama al API Gateway. 
3. API Gateway enruta a Servicio correspondiente. 
4. Servicio valida token con Servicio de Autenticación. 

5. Servicio de Inventario actualiza Base de Datos y publica evento en la Cola. 
6. Servicio  de  Reportes  consume  eventos  para  actualizar  agregados  en 
cache o almacén analítico. 
7. Notificaciones se envían de forma asíncrona.

Preguntas de reflexión: 
Cómo afecta el diseño de la arquitectura a la mantenibilidad y escalabilidad 
de la aplicación 
✓ Mantenibilidad mejora con modularidad: separar responsabilidades en 
servicios y capas (API, lógica de negocio, persistencia) facilita pruebas, 
despliegues y cambios localizados. Código bien documentado y patrones 
claros reducen la deuda técnica. 
✓ Escalabilidad se  facilita  con  separación  de  componentes:  escalar 
horizontalmente la capa de aplicación y cachear consultas frecuentes 
reduce carga en la base de datos. Diseñar límites de contexto claros 
(bounded  contexts)  evita  acoplamientos  que  impidan  escalar  partes 
independientes. 
Qué patrones de diseño orientado a objetos podrían aplicarse en tu diseño 
de arquitectura 
✓ Repository,  Factory,  Strategy,  Unit  of  Work,  Observer,  Adapter, 
Facade. Estos patrones ayudan a desacoplar, facilitar pruebas unitarias, 
soportar múltiples políticas de negocio y conectar con sistemas externos 
sin romper el núcleo del dominio. 
Cómo se pueden abordar los requisitos de seguridad y rendimiento en el 
diseño 
✓ Seguridad:  autenticación  centralizada  (OAuth2/JWT),  autorización  por 
roles, cifrado TLS, validación de entrada, logging de auditoría, pruebas de 
penetración y gestión de secretos. 
✓ Rendimiento: uso de cache (Redis), índices y consultas optimizadas en 
BD,  colas  para  operaciones  asíncronas,  paginación  en  endpoints, 
limitación de tasa (rate limiting) en API Gateway y pruebas de carga para 
dimensionar recursos. 
Consideraciones al migrar esta arquitectura a microservicios 
✓ Descomposición por dominio: identificar bounded contexts (Inventario, 
Compras, Proveedores, Reportes). 
✓ Gestión de datos: cada microservicio con su propia base de datos para 
evitar acoplamientos; usar eventos para sincronización. 
✓ Transacciones  distribuidas:  evitar  transacciones  distribuidas;  usar 
sagas o compensaciones para mantener consistencia. 
✓ Observabilidad: trazabilidad distribuida, métricas y alertas por servicio. 

✓ Operaciones:  automatización  CI/CD,  despliegue  canary,  gestión  de 
configuración y secretos. 
✓ Costos y complejidad: evaluar sobrecarga operativa y decidir si la escala 
y la necesidad de despliegue independiente justifican la migración.
